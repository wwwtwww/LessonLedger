# 产品需求说明书 (PRD) - 全能课时账本 (全球通用版)

## 1. 产品定位与全球愿景

* **产品名称**：课时账本 / LessonLedger
* **产品定位**：全年龄段适用的“全球化全品类课时记账本 + 家庭多端实时同步消课打卡工具”。
* **目标用户**：
    1. **育儿家庭**：需要管理多娃、多门线下培训班（足球、钢琴、美术等）课时与财务账目的家庭。
    2. **终身学习/健身成人**：记录自身健身私教课、瑜伽课、驾校学车、成人乐器、语言培训等消课进度的白领与运动爱好者。
* **核心痛点**：
    1. **跨成员账目混乱**：家庭内部多名成员（大娃、二娃、成人自己）的学费、课时和时间线完全交织，难以统筹管理。
    2. **多端同步滞后**：多名家庭成员（如爸爸交钱、妈妈带娃上课、老人接送）无法实时掌握最新剩余课时，容易发生重复消费或漏打卡。
    3. **机构消课纠纷**：由于请假、缺席、调课，导致培训机构/健身房的消课记录经常与用户心智冲突，迫切需要独立的“第三方财务级记账凭证”防坑。
    4. **全球化多币种需求**：海外多语种家庭或跨国境报课（如国内网课用人民币，本地私教用美元）对界面多语言及多货币原生支持的需求。

## 2. 家庭共享、用户认证与数据恢复架构 (方案 B+)

为了兼顾 C 端首次使用转化率与长期账本数据安全，本应用采用 **Supabase 匿名登录 (Anonymous Sign-in) 起步 + 可选账号绑定恢复 + 租户隔离** 机制。用户首次使用无需注册即可进入；当用户创建家庭账本后，系统引导其绑定 Apple / Google / Email 账号作为可跨设备恢复的身份凭证。家庭邀请码保留为家庭成员加入与备用恢复方式。

### 2.1 用户旅程 (User Flow)
1. **首次打开应用**：系统默默在后台完成 Supabase 匿名登录，用户无需输入邮箱或验证码，直接进入“欢迎页”。
2. **欢迎页三通道选择**：
   * **通道一：【创建新家庭账本】**：系统在云端生成一个全局唯一的 `family_id`，并随机分配一个 6 位大写字母+数字的**家庭邀请码**（如：`A8F9K2`）。该设备自动成为该家庭的“创建者（Creator）”。
   * **通道二：【登录找回我的数据】**：用户通过 Apple / Google / Email 登录。系统根据 Supabase Auth 的用户身份查询 `user_profiles`，若已绑定家庭账本，则自动恢复对应 `family_id` 下的成员、课程与日志数据；若未找到任何已绑定家庭账本，则提示“未找到可恢复数据”，并提供【创建新家庭账本】与【输入邀请码加入】两个后续入口。
   * **通道三：【输入邀请码加入】**：用户输入 6 位家庭邀请码，系统校验成功后，将当前设备账号静默绑定到对应的 `family_id` 下，身份为“成员（Member）”。
3. **账号绑定引导**：用户创建家庭账本后，系统提示其绑定 Apple / Google / Email 账号，确保换手机、卸载重装或设备丢失后仍可找回数据。
4. **数据实时同步 (Real-time)**：一个家庭内的所有设备，通过监听 Supabase Realtime 广播，实现数据（打卡、新增课程等）的秒级跨端同步。

### 2.2 租户隔离安全性 (RLS)
数据库全表开启 **行级安全策略 (Row Level Security)**。所有数据请求在云端强制拦截，只允许读取或操作 `family_id` 匹配当前登录用户档案（`user_profiles`）的数据，从物理层面 100% 杜绝家庭数据外泄。

### 2.3 数据恢复与账号绑定机制

为避免用户换手机、卸载重装或设备丢失后无法找回长期课时数据，系统采用 **“匿名登录起步 + 可选绑定恢复账号 + 家庭邀请码兜底”** 的恢复策略。

1. **匿名登录起步**：首次打开应用时仍使用 Supabase Anonymous Sign-in，保证用户可以零注册体验核心功能。
2. **绑定恢复账号**：创建家庭账本后，系统引导用户绑定 Apple / Google / Email 账号。绑定后，该身份成为用户跨设备恢复数据的主要凭证。
3. **换机自动恢复**：用户在新设备上选择“登录找回我的数据”，使用同一个 Apple / Google / Email 账号登录后，系统查询 `user_profiles.family_id`，自动恢复该家庭账本下的全部数据。
4. **邀请码兜底恢复**：未绑定账号的用户，仍可通过家庭邀请码加入原家庭账本。家庭邀请码同时用于邀请爸爸、妈妈、老人等家庭成员加入同一个账本。
5. **不可恢复边界**：如果用户既未绑定 Apple / Google / Email 账号，也无法获得原家庭邀请码，系统无法验证其数据归属，无法恢复原数据。
6. **数据安全提示**：设置页需常驻展示当前数据安全状态：已绑定账号时显示“数据已开启换机恢复”；未绑定账号时显示“当前仅可通过家庭邀请码恢复，建议绑定账号”。
7. **恢复入口**：欢迎页与设置页均需提供数据恢复相关入口，包括“登录找回我的数据”、“输入家庭邀请码加入”、“复制邀请码”、“分享邀请码”和“绑定恢复账号”。

### 2.4 MVP 账号边界约束

1. MVP 阶段一个用户账号仅允许加入一个家庭账本。
2. 匿名用户绑定 Apple / Google / Email 后，原 `family_id` 保持不变。
3. 如果用户尝试绑定的 Apple / Google / Email 已经关联另一个家庭账本，系统需提示用户选择“进入已有家庭”或“取消绑定”，MVP 阶段不自动合并两个家庭账本。
4. 若用户选择“进入已有家庭”，系统需退出当前匿名家庭上下文并切换到该账号已关联的家庭账本；当前匿名家庭不会自动合并，也不会自动删除。若当前匿名家庭没有其他成员，后续可由用户确认或后台清理任务删除。
5. 家庭邀请码支持在设置页复制和分享。
6. 家庭邀请码泄露时，创建者可在设置页重置邀请码。
7. 删除账号时，仅删除当前用户档案；如果该用户是家庭最后一个成员，需额外确认是否删除整个家庭账本数据。

## 3. 核心数据结构 (Data Schema)

系统采用严谨的四层嵌套关系：`家庭 (Family) ➔ 成员 (Member) ➔ 课程 (Class) ➔ 历史日志 (Log)`。

### 3.1 家庭组实体 (Family)
```json
{
  "id": "String (UUID, 唯一标识)",
  "invite_code": "String (6位唯一随机邀请码，如：A8F9K2)",
  "created_at": "Timestamp (创建时间)"
}
```

### 3.2 用户档案实体 (User Profile)
```json
{
  "id": "String (UUID, 对应 Supabase Auth 的 user_id)",
  "family_id": "String (UUID, 关联所属家庭)",
  "role": "String (角色：creator 创建者 / member 成员)",
  "auth_provider": "String (认证来源：anonymous / apple / google / email，MVP 阶段仅支持这四类)",
  "is_recovery_enabled": "Boolean (是否已绑定可恢复账号)",
  "created_at": "Timestamp",
  "updated_at": "Timestamp"
}
```

> 设置页的数据安全状态由 `auth_provider` 与 `is_recovery_enabled` 共同决定：匿名且未绑定时显示“当前仅可通过家庭邀请码恢复，建议绑定账号”；绑定 Apple / Google / Email 后显示“数据已开启换机恢复”。

### 3.3 成员实体 (Member)
```json
{
  "id": "String (UUID, 唯一标识)",
  "family_id": "String (UUID, 所属家庭ID，用于数据隔离)",
  "name": "String (成员姓名/称呼，如：大娃、妈妈、自己)",
  "type": "String (类型：adult 成人 / kid 孩子，用于未来个性化 UI 策略)",
  "avatar": "String (头像或Emoji标签，如：👶/👧/🏋️/🎹)",
  "themeColor": "String (专属主题颜色十六进制，用于UI区分，如：#3B82F6/#EC4899)"
}
```

### 3.4 课程实体 (Class)
```json
{
  "id": "String (UUID, 唯一标识)",
  "family_id": "String (UUID, 所属家庭ID)",
  "memberId": "String (UUID, 所属成员ID)",
  "name": "String (课程名称，如：少儿编程)",
  "totalPrice": "Number (购买总价)",
  "totalLessons": "Number (购买总课时)",
  "doneLessons": "Number (已消费课时数)",
  "currency": "String (货币符号，如：¥ / $ / €，默认跟随手机系统区域)",
  "unitType": "String (单位类型：lesson 节 / session 次)",
  "duration": "Number (单次课时长，默认60分钟)",
  "schedule": "Array<ScheduleEntry> (结构化排期，格式：{type:'weekly', day:1, time:'18:00'})",
  "notificationIds": "Array<String> (绑定的本地推送通知ID)",
  "isDeleted": "Boolean (软删除标记，默认 false)"
}
```

> MVP 阶段不做汇率换算。每门课程独立保存 `currency`，金额仅在同币种内汇总；跨币种家庭账本只展示分币种小计，不进行跨币种自动折算。

### 3.5 日志流水实体 (Log)
```json
{
  "id": "String (UUID, 唯一标识)",
  "family_id": "String (UUID, 所属家庭ID)",
  "classId": "String (UUID, 关联的课程ID)",
  "type": "String (日志类型：check_in 消课 / skip 请假 / top_up 续费 / init 期初建账 / reversal 冲销撤销)",
  "lesson_delta": "Number (课时变动数量，如打卡记 -1，续费记 +20，期初建账记 -X，冲销记相反数)",
  "money_delta": "Number (金额变动数量，续费为正数，其他日志通常为 0，冲销记相反数)",
  "reversed_log_id": "String (UUID, 仅 reversal 类型必填，指向被冲销的原日志)",
  "timestamp": "ISO 8601 String (发生的真实业务时间，支持用户补卡时手动调整历史日期)",
  "note": "String (选填：备注，如：今天教练请假换人了 / 续费交现金)"
}
```

### 3.6 日志不可变与撤销机制

日志流水采用不可变账本设计。所有已经生成的日志不得物理删除或直接修改；用户撤销误打卡、误请假或误续费时，系统生成一条 `type: 'reversal'` 的冲销日志，用相反的课时/金额变动抵消原日志。

`reversal` 日志必须通过 `reversed_log_id` 关联到被冲销的原始日志，确保系统能够准确判断某条流水是否已撤销，并在 PDF 对账单与历史流水中保留完整审计链路。

## 4. 核心业务流程与 MVP 边界

### 4.1 商业化变现核心：一键对账单导出 (PDF)
* **业务逻辑**：家长可点击任意课程或选定时间段，系统自动聚合该周期内所有的打卡流水（`Log`）。
* **功能输出**：一键生成排版极具专业感的“财务级对账凭证”PDF 报告。
* **分发渠道**：调用系统原生分享直接发送至微信、WhatsApp 或机构老师，作为与培训机构核对不一致课时的第三方证据。
* **商业价值**：此功能作为引导用户购买“29元/3美元”Pro 版的高级买断抓手。

### 4.2 离线优先与极端状态下的“丝滑体验”
* **秒开体验**：所有读取请求优先走本地 `AsyncStorage` 缓存，实现用户开屏秒开。
* **乐观更新 (Optimistic Updates)**：打卡消课、修改数据时，界面立即完成扣减并触发触感反馈（Haptics），后台异步与 Supabase 通信，杜绝因网络延迟造成的卡顿。
* **断网队列**：网络彻底断开时，请求进入本地 `Sync Queue` 队列，恢复网络后在后台静默重放。
* **首次断网边界**：如果用户首次打开 App 时无网络，系统无法完成匿名登录与家庭账本创建，应展示离线提示页，引导用户联网后继续。若用户此前已登录且本地存在缓存数据，则允许直接进入离线模式，所有写操作进入本地同步队列。
* **失败回滚策略**：乐观更新失败时，系统需自动回滚本地状态，并展示明确提示，如“同步失败，已恢复为上一次有效状态”。

### 4.3 核心高频真实场景：期初建账 (Opening Balance)
* **解决痛点**：解决新用户半路开始用 App 时，无法或嫌麻烦不愿倒推过去历史打卡的问题。
* **业务实现**：在“新增课程”表单中提供【已上课时】输入框（默认为 0）。用户若输入 15，系统在创建课程时自动设置 `doneLessons = 15`，并静默生成一条 `type: 'init', lesson_delta: -15` 的日志。首页立即呈现正确的剩余课时，甩掉历史包袱。

### 4.4 核心高频真实场景：原链续费 (Top-up)
* **解决痛点**：旧课程上完后续费，不需要重新建立一个新课程卡片（导致历史数据割裂）。
* **业务实现**：点击课程卡片的【续费】按钮，输入新增的续费金额、续费课时。系统执行累加：`totalPrice = totalPrice + 新金额`, `totalLessons = totalLessons + 新课时`。同时生成一条 `type: 'top_up', lesson_delta: +新课时, money_delta: +新金额` 的日志流水，历史账目连续可追溯。

## 5. 边缘场景：防呆与容错

* **时间冲突硬拦截**：给同一个成员设置两个在时间轴和单次时长（`duration`）上重叠的排期时，系统进行拦截弹窗提醒，防止漏报或排课失误。
* **忘打卡日历补签**：支持用户在打卡时通过日历面板重选真实的过去日期，确保消课日志的时间戳能够完美对应学校的真实点名册。
* **触感撤销条 (Undo)**：用户点击打卡后，底部弹出类似 iOS 提醒事项的“撤销提示条”，带有 5 秒倒计时。倒计时内点击一键撤销，系统生成 `type: 'reversal'` 的冲销日志并回滚课时计数，修复误触。
* **纯记录请假 (Skip)**：提供“请假”专用打卡按钮。触发后不扣减实际剩余课时，但会生成一条 `type: 'skip'` 带有备注（如：孩子感冒请假）的时间轴日志，便于核对机构为什么多扣了课时。
* **超额打卡拦截**：当课程剩余课时小于本次消课数量时，MVP 阶段默认禁止打卡，并提示用户先进行续费。未来可扩展为课程级“允许超额消课”高级设置。

## 6. 关键技术落地对策与技术排坑 (研发落地方案)

为了确保“一人公司”研发模式下的高效与底层系统稳健，实施时须严格遵守以下四项技术对策：

### 6.1 开发阶段的渐进式删库重建策略 (Clean Slate DDL)
* **研发共识**：鉴于应用处于早期闭环研发阶段，无任何线上真实活跃用户。为了杜绝维护复杂向后兼容迁移脚本（Migration）引发的精力消耗，开发期间一律不保留旧版本的本地与云端测试数据。
* **落地对策**：采用“清空重建（Push-to-Reset）”策略。每次对底层 Data Schema 进行大修（如引入多租户账户体系）时，一律通过完整的 SQL 脚本重建云端数据库表（先 `DROP TABLE IF EXISTS`），配合前端 AsyncStorage 清理逻辑，使项目在最干净、无污染的字段结构中高速迭代。

### 6.2 服务端去重防碰撞邀请码 (Anti-Collision)
* **技术挑战**：6 位大写字母+数字共有约 21 亿种组合，在高并发生成家庭邀请码时，前端随机生成极易遭遇“碰撞冲突”，导致创建失败。
* **落地对策**：将生成压力和校验全量移交至服务端。在 Supabase 编写一个 PostgreSQL 存储过程（RPC 自定义函数 `generate_unique_invite_code()`），利用云端 `LOOP` 循环生成并直接查询校验唯一性，碰到冲突自动重试，前端只需静默调用接口，100% 确保原子性与并发安全。

### 6.3 事件溯源级解决多端同步离线冲突 (Conflict Resolution)
* **技术挑战**：多端异步操作高发（如爸爸在弱网地下室打卡扣减课时，妈妈在家同时打卡）。直接修改并覆盖 `classes.doneLessons` 会导致先提交的数据被后到达的数据覆盖。
* **落地对策**：底层采用**“事件溯源 (Event Sourcing)”**架构理念。移动端写操作时，严格禁止直接 update 课程的剩余数，而是统一向数据库 `logs` 表中插入一条独立的增量流水记录（如打卡记 `lesson_delta: -1`）。云端配置 PostgreSQL 触发器 (Trigger)，捕捉到 `logs` 表有新插入记录时，自动执行累加聚合，更新 `classes.doneLessons`。通过流水的叠加彻底解决多端并发冲突。

### 6.4 基于系统原生内核的 HTML-PDF 矩阵 (PDF Export)
* **技术挑战**：React Native 跨平台直接绘制原生 Canvas 并生成复杂 PDF 布局在 Android 和 iOS 两端极易出现排版错位、中文字体缺失等不兼容隐坑。
* **落地对策**：采用 `expo-print` 配合内置 HTML 渲染引擎。前端将日志流水数据与封装好的 Tailwind CSS / 内联样式的 HTML 模板字符串进行动态拼接，直接交由手机系统底层的 WebView 打印内核渲染输出，保证双端呈现像素级对齐的高质量财务对账单 PDF。

## 8. 重构实施分层 (Implementation Phases)

当前 PRD 面向 MVP 后的系统重构，目标不是压缩产品能力，而是将账号安全、家庭共享、可信账本、离线同步与商业化能力拆分为可控的施工阶段，避免一次性实现所有复杂模块导致研发失控。

### Phase 1：数据安全与家庭账本底座

优先建立新的多租户数据底座，确保家庭账本的数据归属、隔离和基础同步稳定。

* Supabase 匿名登录起步。
* Family / User Profile / Member / Class / Log 核心表结构。
* 基于 `family_id` 的 RLS 租户隔离。
* 创建家庭账本与生成家庭邀请码。
* 通过家庭邀请码加入已有家庭。
* 成员、课程、日志的基础云端读写。
* 设置页展示家庭邀请码与基础数据安全状态。

### Phase 2：账本可信度与恢复能力

在基础数据底座稳定后，补齐长期账本产品必须具备的数据恢复和审计链路能力。

* Apple / Google / Email 账号绑定恢复。
* “登录找回我的数据”流程。
* 绑定冲突处理：进入已有家庭 / 取消绑定。
* 设置页展示完整数据安全状态。
* 日志不可变机制。
* `reversal` 冲销撤销机制。
* 期初建账、续费、请假、补签。
* 超额打卡拦截。

### Phase 3：多端体验与离线增强

在账本语义和账号恢复稳定后，再增强多端实时性与弱网体验，避免在底层模型未稳定前引入复杂同步问题。

* AsyncStorage 本地缓存秒开。
* 乐观更新与失败回滚。
* Supabase Realtime 多端秒级同步。
* Sync Queue 断网写入队列。
* 网络恢复后的后台静默重放。
* 多端并发写入下的冲突处理与幂等保护。

### Phase 4：商业化与高级能力

在可信账本和多端同步闭环稳定后，再推进商业化抓手和高级体验能力。

* PDF 对账单导出。
* Pro 买断付费墙。
* 本地通知与课前提醒。
* 排期冲突硬拦截。
* Dashboard V2 数据可视化。
* 多语言与多地区商业化策略优化。
