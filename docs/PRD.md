# 产品需求说明书 (PRD) - 全能课时账本 (全球通用版)

## 1. 产品定位与全球愿景

* **产品名称**：时课账本 / LessonLedger
* **产品定位**：全年龄段适用的“全球化全品类课时记账本 + 高效消课打卡工具”。
* **目标用户**：
    1. **育儿家长**：管理多娃、多门线下培训班（足球、钢琴、美术等）课时与账目的家庭。
    2. **终身学习/健身成人**：记录自身健身私教课、瑜伽课、驾校学车、成人乐器、语言培训等消课进度的白领与运动爱好者。
* **核心痛点**：
    1. **跨成员账目混乱**：家庭内部多名成员（大娃、二娃、成人自己）的学费、课时和时间线完全交织，难以统筹管理。
    2. **机构消课纠纷**：由于请假、缺席、调课，导致培训机构/健身房的消课记录经常与用户心智冲突，迫切需要独立的“第三方记账凭证”防坑。
    3. **多语种需求**：海外用户或跨国籍家庭对多语言界面的原生支持需求。

## 2. 核心数据结构 (Data Schema)

为了支撑“多成员+多语种+全品类”的底层业务，系统采用四层嵌套关系：`家庭 (Family) ➔ 成员 (Member) ➔ 课程 (Class) ➔ 打卡日志 (Log)`。

### 2.1 成员实体 (Member)
```json
{
  "id": "String (唯一标识)",
  "name": "String (成员姓名/称呼，如：大娃、妈妈、自己)",
  "type": "String (类型：adult / kid，用于未来个性化UI推荐)",
  "avatar": "String (头像或Emoji标签，如：👶/👧/🏋️/🎹)",
  "themeColor": "String (专属主题颜色十六进制，用于UI高亮区分，如：#3B82F6/#EC4899)"
}
```

### 2.2 课程实体 (Class)
```json
{
  "id": "String (唯一标识)",
  "memberId": "String (所属成员ID)",
  "name": "String (课程名称，如：少儿编程)",
  "totalPrice": "Number (购买总价)",
  "totalLessons": "Number (购买总课时)",
  "doneLessons": "Number (已上课时数)",
  "unitType": "String (单位类型：lesson/session)",
  "duration": "Number (单次课时长，默认60分钟)",
  "schedule": "Array<ScheduleEntry> (结构化排期)",
  "notificationIds": "Array<String> (绑定的本地推送ID)",
  "isDeleted": "Boolean (软删除标记)"
}
```

## 3. 核心业务流程与 MVP 边界

### 3.1 离线优先与极端状态下的“丝滑体验”
为了确保家长在电梯里、健身房地下室等弱网环境下依然能顺畅使用：
- 所有数据请求走 `AsyncStorage` 缓存以实现**秒开**。
- 用户点击“打卡”等写操作时，采用**带互斥锁的乐观更新**（界面先变，后台提交），彻底杜绝由于网络超时导致的 UI 卡死。
- 网络请求失败的操作统一进入 `Sync Queue`（同步队列），在后台静默重放，不打断用户操作心智。

### 3.2 边缘场景：防呆与容错
- **时间冲突**：当家长给同一个孩子设置了两个时间重合的补习班时，系统底层基于 `duration` 和 `schedule` 字段计算出跨天重叠或跨时区偏移，进行**硬性拦截警告**。
- **忘打卡补签**：每日系统自动嗅探昨日未打卡的课程。支持用户通过日历重选真实日期，确保日志（Log）的时间戳可以与机构的请假签到本完美对账。
- **误触撤销**：打卡后底部提供临时展示的“撤销条（Undo）”，在倒计时结束前一键回退 `doneLessons`，并恢复系统本地闹钟。
- **记录请假**：提供“请假（Skip）”通道，只产生备注日志，不扣减实际课时，方便记录“为什么这周没去”。

## 4. 未来规划 (V2 Phase)
在资产记账、通知提醒和防出错的闭环打通后，下一个战略重点是**数据可视化赋能**：
- [ ] 增加报表（Dashboard V2）页面，包括按成员归类的费用分布饼图。
- [ ] 呈现未来一个月与过去三个月的消课趋势图（Line Charts），形成正向的打卡激励反馈环。
